> **TL;DR**
> - Raspberry Pi 5 (16GB) で完全オンプレのRAGナレッジ基盤を構築
> - 構成: FastAPI + Ollama (Gemma3 7B) + ChromaDB + Redis（Docker 4コンテナ）
> - 月額コスト: 電気代のみ（約100円/月）
> - RAG応答速度: 約8秒（量子化で3〜4秒）
> - クラウドAPI代を減らしたい・データを外に出したくないエンジニア向け

## クラウドAI、高くない？

「GPT-4oのAPI代、今月いくらだっけ…」って明細見て冷や汗かいたことない？

なのらもそうだった。毎月じわじわ増えていくAPIコスト、しかも自分のドキュメントやIssueのデータを全部外部に送ってるのがずっと気になってた。

そこで**Raspberry Pi 5を使って、完全オンプレで動くAIナレッジ基盤**を作ってみた。クラウド費用ゼロ、データは手元に、24時間自律稼働。これがかなり実用レベルになってきたので共有します！

---

## Raspberry Pi 5、意外といけるんです

「ラズパイでAI？ 無理でしょ」って思うよね。あたしも最初そうだった。

でも**Raspberry Pi 5（16GB）**は一世代前とは別物。

| スペック | 実用上の意味 |
|---------|-------------|
| ARM Cortex-A76 × 4コア | 7Bモデルの推論を実用速度でさばける |
| 16GB LPDDR4X | ベクトルDB + LLM同時起動でも安定 |
| PCIe 2.0（NVMe対応） | microSDの10倍速I/O |
| 消費電力 5〜15W | 24時間つけっぱなしで月の電気代100円以下 |

あたしはPironman 5-MAXケースと組み合わせてるんだけど、ファン制御が優秀で夏場でもサーマルスロットリングなし。ここ大事です！

## ナレッジ取り込み：放っておけば勝手に賢くなる

「毎日手動で情報集めて整理して…」ってやってられないですよね。あたしもそれが嫌で自動化しました。

GitHubのIssue、技術ブログのRSSフィード、Slackのログを毎日自動で収集→チャンク分割→ベクトル化してくれます。

```python
async def ingest_document(content: str, source: str, metadata: dict):
    chunks = text_splitter.split_text(content)

    # Gemini APIでembedding生成（無料枠で十分）
    embeddings = await embed_texts(chunks)

    collection.add(
        documents=chunks,
        embeddings=embeddings,
        metadatas=[{**metadata, "source": source}] * len(chunks),
        ids=[f"{source}_{i}" for i in range(len(chunks))]
    )
```

**ポイント**: embeddingだけはGemini APIの無料枠を使ってる。ローカルLLMでembeddingすると遅すぎるので、なのら的にはここの割り切りが大事だと思ってます。

## RAG質問応答：自分専用の知識ベースに聞ける

溜まったナレッジに対して自然言語で質問できるようにしてあります。

```python
async def query_knowledge(question: str, top_k: int = 5) -> str:
    q_embedding = await embed_text(question)

    results = collection.query(
        query_embeddings=[q_embedding],
        n_results=top_k
    )

    context = "\n---\n".join(results["documents"][0])
    prompt = f"""以下のナレッジを参考に質問に答えてください。

ナレッジ:
{context}

質問: {question}"""

    response = await ollama_client.generate(
        model="gemma3:7b",
        prompt=prompt
    )
    return response["response"]
```

ここで地味に大事なのが**重複排除**。同じ内容のドキュメントが複数ヒットするとコンテキストがノイズまみれになる。取り込み時にJaccard類似度でフィルタリングしてるのがけっこう効いてます。

## ハマったポイント3つ

同じことやろうとしてる人がぶつかりそうな壁を先に共有しておきます。

### 1. メモリ管理、油断するとOOMで死ぬ

16GBあっても、ChromaDB + Ollama + FastAPIの同時起動は余裕がない。**各コンテナにメモリ上限を設定しないとOOMキラーに殺される**。あたしはこれで2回本番データ飛ばしかけました。

```yaml
services:
  agent:
    mem_limit: 2g
  ollama:
    mem_limit: 10g
  chromadb:
    mem_limit: 2g
```

### 2. microSDでは実用にならない

ChromaDBの書き込みがmicroSDだと致命的に遅い。NVMe SSDに変えたら体感10倍。これは最初からNVMe前提で組んだほうがいいです。

### 3. Gemini APIキーは複数持っておく

embedding用の無料枠、1キーだとすぐレート制限に引っかかる。あたしは3キー体制でフォールバックするようにしてて、1つが制限されても止まらない構成にしてます。

## パフォーマンス実測値

| 処理 | 時間 | 体感 |
|------|------|------|
| ドキュメント取り込み（1000チャンク） | 約45秒 | バックグラウンドなので気にならない |
| ベクトル検索（top_k=5） | 約80ms | 体感即座 |
| RAG応答生成（Gemma3 7B） | 約8秒 | ちょっと待つけど許容範囲 |
| 日次サマリー生成 | 約2分 | 寝てる間に終わってる |

RAGの8秒が気になるなら量子化モデル（Q4_K_M）で3〜4秒まで縮まる。あたしは精度優先でフルモデル使ってますが、ここはお好みで。
