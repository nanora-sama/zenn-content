# Raspberry Pi 5で作る自己完結型AIナレッジ基盤

こんにちは、のらちゃんです！

最近「クラウドのAIサービス高すぎ…」「自分のデータを外に出したくない」って思うことない？

そこで今回は、**Raspberry Pi 5を使って完全オンプレで動くAIナレッジ基盤**を構築した話をします。クラウド費用ゼロ、データは手元に、しかも24時間自律稼働。かなり実用レベルになってきたので、実践的にまとめていくよ！

---

## なぜRaspberry Pi 5なのか

正直、最初は「ラズパイでAIとか無理でしょ」って思ってた。でも**Raspberry Pi 5（16GB）**はかなり別格なんです。

- **ARM Cortex-A76 × 4コア**：推論タスクを余裕でさばける
- **16GB LPDDR4X**：ベクトルDBとLLMを同時起動しても安定
- **NVMe SSD対応**：PCIe 2.0経由で高速I/O
- **消費電力 5〜15W**：24時間稼働してもほぼ電気代タダ

自分はPironman 5-MAXケースと組み合わせてるんだけど、温度管理が完璧で夏でも安定してる。

## コアコンポーネント1：ナレッジ取り込みパイプライン

一番大事なのが**ナレッジの自動取り込み**。毎日GitHubのIssue、Slack、技術ブログを自動収集してベクトル化してる。

```python
# knowledge_ingester.py（抜粋）
async def ingest_document(content: str, source: str, metadata: dict):
    # テキストをチャンク分割
    chunks = text_splitter.split_text(content)
    
    # Gemini APIでembedding生成（無料枠で十分）
    embeddings = await embed_texts(chunks)
    
    # ChromaDBに保存
    collection.add(
        documents=chunks,
        embeddings=embeddings,
        metadatas=[{**metadata, "source": source}] * len(chunks),
        ids=[f"{source}_{i}" for i in range(len(chunks))]
    )
```

ポイントは**Gemini APIのembedding**を使ってること。ローカルLLMでembeddingすると遅いので、ここだけクラウドを使う割り切りが重要。

## コアコンポーネント2：RAGベースの質問応答

```python
async def query_knowledge(question: str, top_k: int = 5) -> str:
    # 質問をベクトル化
    q_embedding = await embed_text(question)
    
    # 類似度検索
    results = collection.query(
        query_embeddings=[q_embedding],
        n_results=top_k
    )
    
    # コンテキスト付きでローカルLLMに投げる
    context = "\n---\n".join(results["documents"][0])
    prompt = f"""以下のナレッジを参考に質問に答えてください。

ナレッジ:
{context}

質問: {question}"""
    
    # Ollama (Gemma3)でローカル推論
    response = await ollama_client.generate(
        model="gemma3:7b",
        prompt=prompt
    )
    return response["response"]
```

検索精度を上げるためにJaccard類似度でのフィルタリングも入れてる。重複ドキュメントが混入するとノイズになるから、取り込み時の重複排除が地味に重要。

## コアコンポーネント3：スケジューラーによる自律収集

手動で何もしなくても、毎日自動でナレッジが更新されるようにしてある。

```python
# scheduler.py（抜粋）
scheduled_tasks = [
    {"cron": "0 3 * * *", "task": "ingest_github_issues"},
    {"cron": "0 6 * * *", "task": "ingest_tech_feeds"},
    {"cron": "30 9 * * *", "task": "generate_daily_summary"},
]
```

深夜3時にGitHubのIssueを全部食わせて、朝6時に技術ブログを収集、9時半には日次サマリーが自動生成されてTelegramに届く。起きたら昨日の重要情報がまとまってる感じ、めっちゃ便利。

## 実際の運用で気づいた落とし穴

### 1. メモリ管理が超重要

16GBあっても、ChromaDB + Ollama + FastAPIを同時起動すると余裕がなくなる。各コンテナにメモリ上限を設定して、OOMキラーに殺されないようにすること。

```yaml
# docker-compose.yml
services:
  agent:
    mem_limit: 2g
  ollama:
    mem_limit: 10g
  chromadb:
    mem_limit: 2g
```

### 2. NVMe SSDは必須

microSDだとChromaDBの書き込みが遅すぎて実用にならない。NVMeにしたら体感で10倍速くなった。

### 3. Gemini API無料枠の使い方

embeddingだけならGemini API無料枠で十分なんだけど、**APIキーは複数持っておく**のがおすすめ。自分は3キー体制でフォールバックするようにしてて、レート制限に引っかかっても止まらない。

## パフォーマンス実測値

| 処理 | 時間 |
|------|------|
| ドキュメント取り込み（1000チャンク） | 約45秒 |
| ベクトル検索（top_k=5） | 約80ms |
| RAG応答生成（Gemma3 7B） | 約8秒 |
| 日次サマリー生成 | 約2分 |

RAGの応答8秒はちょっと遅いけど、非同期で走らせてるのでUXとしては許容範囲。もっと速くしたいなら量子化モデル（Q4_K_M）を使うと3〜4秒になる。
